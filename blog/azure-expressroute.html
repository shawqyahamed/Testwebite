Optimizing Cloud Connectivity with Azure ExpressRoute

📅 September 2023 ⏱ 6 min read 👤 Shawqy Ahamed

When an enterprise commits to Azure as its cloud platform, the connectivity between on-premises infrastructure and Azure becomes a critical path dependency. Internet-based connections (VPN Gateway) work for low-bandwidth, latency-tolerant workloads, but for SAP HANA, SQL Managed Instance, or latency-sensitive financial applications, you need dedicated connectivity — and that means ExpressRoute.

ExpressRoute delivers consistent sub-10ms latency and predictable bandwidth that no Internet-based connection can match. For workloads where performance SLAs matter, it is not optional.

ExpressRoute Fundamentals

ExpressRoute provides a private, dedicated connection between your on-premises network and Microsoft's global network, bypassing the public Internet entirely. The connection is established through an ExpressRoute partner — a telco or network service provider — at one of Microsoft's peering locations. Your traffic never touches the public Internet between your network and Azure.

There are two peering types you'll typically configure: Azure Private Peering provides connectivity to Azure virtual networks and services deployed within VNets. Microsoft Peering provides connectivity to Microsoft public services like Microsoft 365, Dynamics 365, and Azure public endpoints.

Designing for Redundancy

A single ExpressRoute circuit, even with its built-in redundancy (each circuit has two physical paths), is not sufficient for business-critical workloads. The correct architecture for 99.99% availability is dual circuits from different providers, connecting to different ExpressRoute peering locations, with BGP configured for automatic failover.

BGP Configuration for Active-Active

Configure both circuits in active-active mode with equal BGP weight and local preference. Use AS-PATH prepending to prefer one circuit for outbound traffic while maintaining the other as an active hot standby. This gives you immediate failover — typically under 30 seconds — without any manual intervention.

  • Primary circuit: Default BGP weight (100), no AS-PATH prepending
  • Secondary circuit: Default BGP weight (100), AS-PATH prepend x1 for outbound preference
  • Configure BFD (Bidirectional Forwarding Detection) for sub-second failure detection
  • Set hold-time to 90 seconds and keepalive to 30 seconds on BGP sessions

Performance Optimization

Once the circuits are up, focus on traffic engineering. Use ExpressRoute Global Reach to connect your on-premises sites to each other through Microsoft's backbone — often faster than your own WAN for intercontinental traffic. Enable FastPath on your ExpressRoute Gateway to bypass the gateway for data-path traffic, reducing latency for high-throughput workloads.

Monitor circuit utilization carefully. ExpressRoute circuits come in fixed bandwidths (50Mbps to 100Gbps). Sustained utilization above 70% on a circuit indicates you need to either upgrade the circuit or implement traffic engineering to shift flows to the secondary circuit.

The Results

In the Azure WAN optimization project I led, the combination of dual ExpressRoute circuits, BGP route optimization, and FastPath configuration delivered a 40% reduction in average round-trip latency for workloads running in Azure. More importantly, we achieved 99.99% connectivity uptime over 18 months — with two transparent failover events that users never noticed.
