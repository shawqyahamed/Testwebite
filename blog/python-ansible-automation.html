Automating Network Operations with Python & Ansible

📅 August 2023⏰ 5 min read👤 Shawqy Ahamed

Network engineers spend an enormous amount of time on repetitive, error-prone manual tasks: pushing configuration changes to hundreds of devices, collecting interface statistics, verifying ACL consistency, generating compliance reports. When I joined SecureNet Solutions, the team was spending 15+ hours every week on these tasks. By the time I left, that number was under 2 hours — and the remaining time was spent reviewing automated outputs, not running commands manually.

Automation doesn't replace network engineers — it frees them to do the work that actually requires human judgment: design, troubleshooting, and strategic planning.

The Automation Stack

The toolchain I settled on after evaluating several options: Python with Netmiko for device-level connectivity and configuration push, NAPALM for multi-vendor abstraction and validation, Ansible for orchestrating multi-device workflows and integrating with CI/CD pipelines, and Jinja2 for templating device configurations from structured data.

Why Netmiko over Paramiko?

Paramiko is the underlying SSH library, but Netmiko wraps it with network-device-specific handling: it knows how to handle Cisco's "More" prompts, Juniper's commit workflow, and device-specific timing requirements. Writing this yourself with raw Paramiko is possible but tedious. Netmiko handles it out of the box for 50+ network platforms.

Configuration Management with Templates

The most impactful automation we built was configuration templating. Instead of engineers writing device configurations by hand, we built Jinja2 templates for every standard configuration pattern — access switch, distribution switch, WAN router, firewall policy. Network data (VLANs, IP addressing, routing parameters) was stored in YAML files in a Git repository.

When a new site needed to be brought up, the engineer updated the YAML data file, ran the Ansible playbook, and the configuration was generated, validated against a set of compliance checks, and pushed to all devices in the site — all in under 10 minutes. The same process previously took a full day of manual work.

Automated Compliance Checking

One of the most valuable automation use cases is compliance validation. We wrote a Python script that connects to every device in the network, collects the running configuration, and compares it against our defined standards using regular expressions and structured parsing with TextFSM. Any device that deviates from the standard generates an alert in our ITSM system.

  • NTP server configuration (must use corporate NTP servers)
  • SSH version (must be SSHv2 only)
  • Password encryption (must use Type 8 or Type 9)
  • AAA configuration (must authenticate against ISE)
  • SNMP community strings (no default strings, must use SNMPv3)

Getting Started

If you're new to network automation, start small. Pick one repetitive task — maybe it's collecting interface error statistics from 20 switches — and write a Python script to do it. Once you've seen the value, expand from there. The key is to build automation iteratively, test thoroughly in a lab environment, and treat your automation code with the same discipline as application code: version control, code review, and testing.

The 15 hours we saved every week compounded into something much larger: fewer configuration errors, faster site deployments, better compliance posture, and a team that had time to think strategically rather than operate reactively.
